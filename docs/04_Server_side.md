# 04 — Server-side: Chunked Storage, Per-Chunk DEK Encryption and Key Rotation

This document describes the **server-side implementation** of the Secure Cloud Storage system (fourth version): chunked storage, per-chunk DEK encryption, re-encryption and key rotation.

## 1. Requirements covered

### 1.1 Lab and product requirements

| Requirement |
|-------------|
| **Divide files into chunks and store them encrypted with different DEKs**: Files are split into 1MB chunks, each encrypted with a unique DEK generated by the KMS |
| **Implement data re-encryption from an old key**: After a key rotation, all SSE files are re-encrypted chunk by chunk with new DEKs wrapped by the new Master Key |
| **Keep old Master Keys for decryption of old files**: Old Master Keys are archived (status: "old") in the KMS and can still be used to unwrap DEKs from files encrypted before the rotation |

## 2. Chunked Storage Architecture

| Component | Description |
|-----------|-------------|
| Chunk size | 1MB (1,048,576 bytes) |
| Chunk files | `<file_id>_chunk_0.blob`, `<file_id>_chunk_1.blob`, ... |
| Metadata file | `<file_id>.meta` — stores filename, mode, algorithm, num_chunks and per-chunk DEK info |
| DEK per chunk | Each chunk has its own `wrapped_dek_hex` and `key_version` stored in `.meta` |

### 2.1 Disk structure

```
file_bin/
  <user_id>/
    <file_id>_chunk_0.blob   ← encrypted with DEK_0
    <file_id>_chunk_1.blob   ← encrypted with DEK_1
    <file_id>_chunk_2.blob   ← encrypted with DEK_2
    <file_id>.meta           ← chunk_metas: [{wrapped_dek_hex, key_version}, ...]
  shared/
    <folder_id>/
      <file_id>_chunk_0.blob
      <file_id>.meta
```

### 2.2 Metadata structure (`.meta`)

```json
{
  "filename": "example.txt",
  "encryption_mode": "sse",
  "algorithm_mode": "aesgcm",
  "num_chunks": 3,
  "chunk_metas": [
    {"wrapped_dek_hex": "abc123...", "key_version": 1},
    {"wrapped_dek_hex": "def456...", "key_version": 1},
    {"wrapped_dek_hex": "ghi789...", "key_version": 1}
  ]
}
```

## 3. Implementation

### 3.1 Storage Backend

- **Role**: Manages chunked file storage with per-chunk DEK encryption in SSE mode.
- **Location**: `src/secure_cloud_storage/storage/backend.py`
- **Main operations:**
    - `_split_chunks(data)`: Splits raw bytes into a list of 1MB chunks.
    - `upload(token, file_id, data, ...)`: In SSE mode, splits data into chunks, generates a new DEK per chunk via `KMS.generate_dek()`, encrypts each chunk, and stores `wrapped_dek_hex` and `key_version` in `.meta`. In CSE mode, stores data as a single chunk (already encrypted by the client).
    - `download(token, file_id, ...)`: Reads all chunks, unwraps each chunk's DEK via `KMS.unwrap_dek()`, decrypts and verifies AAD per chunk, and reassembles the original file.
    - `delete(token, file_id, ...)`: Removes all chunk blobs and the `.meta` file.
    - `reencrypt_file(token, file_id, ...)`: Re-encrypts all chunks of an SSE file with new DEKs after a key rotation. Decrypts each chunk with the old DEK, generates a new DEK, re-encrypts and overwrites the chunk on disk, and updates `.meta` with the new DEK info.

### 3.2 Client (application layer)

- **Location**: `src/secure_cloud_storage/client/service.py`
- **Main operations:**
    - `rotate_key(token)`: Calls `KMS.rotate_master_key()` to generate a new Master Key version and archive the old one.
    - `reencrypt_all_files(token)`: Iterates over all personal SSE files and calls `storage.reencrypt_file()` for each one. Returns a summary with reencrypted and failed files.

### 3.3 CLI (Click)

- **Entry:** `uv run python -m secure_cloud_storage`
- **Added command:** `rotate-key` — rotates the Master Key and re-encrypts all SSE personal files. Asks for confirmation before proceeding and displays a summary of re-encrypted files.

```
uv run python -m secure_cloud_storage rotate-key
```

### 3.4 UI (Streamlit)

- **Entry:** `uv run python -m secure_cloud_storage --ui`
- **New behaviour:** A **"🔄 Rotate Master Key"** button has been added under the **Key Management** section in the sidebar. Clicking it re-encrypts all SSE files and rotates the Master Key. A success or error message is displayed after the operation.

## 4. Security considerations

- Each chunk has its own independent DEK, so compromising one DEK only exposes 1MB of data.
- DEKs are never stored in plaintext — they are always wrapped by the user's active Master Key.
- After re-encryption, old DEKs are replaced by new ones wrapped with the new Master Key.
- Old Master Keys are kept archived in the KMS to allow decryption of files encrypted before the rotation.
- Raw DEKs are securely zeroed from memory after use via `secure_zero()`.
